Что реально полезно сделать “прямо сейчас” для серьёзного проекта (и без переписывания URL):

## 1) Удаление меню и удаление пункта меню в UI + API

Сейчас в `menu.php` **нет кнопки “Удалить меню”**, а в `api.php` **нет action `menu.delete`**.
Для проекта это критично: мусорные меню будут копиться, тестовые тоже.

**Что делаем:**

* UI: добавить кнопку **“Удалить меню”** рядом с “Переименовать”.
* API: добавить экшен `menu.delete` (EDITOR+) — удалить меню целиком по `menuId`.

---

## 2) “Переключатель верхнего меню” + защита от битого `topMenuId`

Ты уже сделал “Сделать верхним”. Надо ещё:

* если удалили верхнее меню → `topMenuId` должен обнулиться
* если `topMenuId` указывает на несуществующее меню → игнорировать и показывать “не выбрано”

Это прям “серьёзность” данных.

---

## 3) Логи и аудит (минимальный)

Добавить в записи:

* `updatedAt`, `updatedBy` для меню и menu items
* и/или простой `audit.log` в `/upload/sitebuilder/audit.log` (append) — кто что делал: create/delete/update/move

Это поможет при разборе проблем.

---

## 4) В pages-диалоге (index.php) — сделать “Домашняя” кнопку понятной

У тебя API уже есть: `site.setHome`, но в UI я не вижу нормального управления дом. страницей.

**Что делаем:**

* В диалоге “Страницы” добавить кнопку у каждой страницы: **“Сделать главной”**
* И рядом индикатор “Главная” на текущей.

Это улучшает UX и логику сайта *сразу*.

---

## 5) Валидации в UI (чтобы не создавать “битые” данные)

Пример:

* slug: запрещаем пробелы, русские, “/”, “?” (или хотя бы предупреждение)
* url в кнопках/меню: подсветка если не `http(s)` и не `/...`

Самый правильный следующий шаг: **(1) удаление меню + (2) защита topMenuId** — это инфраструктура данных, без неё проект быстро “загниёт”.
